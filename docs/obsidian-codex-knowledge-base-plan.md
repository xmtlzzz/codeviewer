# Obsidian + Codex 个人知识库方案

## 目标

把 Obsidian 作为知识存储层，把 Codex 作为加工层，把 skills 作为固定流程层，形成一个本地优先、可持续维护的个人知识库。

核心不是“让 AI 替代笔记”，而是把 AI 放在三个最值钱的环节上：

1. 收集时，快速结构化
2. 复盘时，快速归纳和补链
3. 项目推进时，快速把碎片知识变成可执行材料

---

## 推荐结论

最稳的实现方向是：

- `Obsidian` 负责原始笔记、双链、目录和检索
- `Codex` 负责摘要、重写、分类、链接建议、任务生成
- `skills` 负责固定动作的标准化，比如“摘录转笔记”“周复盘”“项目汇总”
- `MCP / 外部连接` 负责接入 GitHub、网页、文档、日历等外部信息源

不要把 Codex 当成唯一存储，也不要把知识库做成一个重数据库系统。知识库的主文件应该仍然是 Markdown。

---

## 三种可行路线

### 1. 轻量版

适合先跑通习惯。

- Obsidian 里只维护 Markdown
- 用模板、标签、双链做组织
- Codex 只在需要时参与整理

优点：

- 成本最低
- 迁移容易
- 不依赖复杂工具链

缺点：

- 自动化弱
- 规模大了后检索和归纳效率一般

### 2. 实用版

适合长期使用。

- 做 3 到 5 个固定 skills
- 每个 skill 对应一个明确动作
- 输出统一写回 Obsidian

优点：

- 形成稳定习惯
- 操作可复制
- 很适合个人工作流

缺点：

- 需要一开始设计好流程
- skill 边界要清楚

### 3. 高阶版

适合笔记很多、检索要求高的情况。

- Obsidian 仍是主库
- 额外做全文索引或语义索引
- Codex 通过索引读取相关笔记，再生成结构化结果

优点：

- 检索能力强
- 适合大规模知识库

缺点：

- 复杂度明显更高
- 维护成本更高

## 我建议的默认架构

### Vault 分层

```text
vault/
├── 00_inbox/         # 临时收集
├── 10_sources/       # 来源资料、摘录、网页笔记
├── 20_atomic/        # 原子知识，单一概念
├── 30_synthesis/     # 主题归纳、长文总结
├── 40_projects/      # 项目笔记、执行计划
├── 50_people/        # 人物笔记
├── 60_systems/       # 方法、流程、模板
└── 90_archive/       # 归档
```

### 笔记规则

- 一条笔记只解决一个问题
- 原始来源和你的理解分开写
- 不要靠大量标签表达层次
- 双链优先于手工复制
- 每条笔记都尽量带最小 frontmatter

推荐字段：

```yaml
---
type: atomic | source | synthesis | project | people | system
status: inbox | active | parked | done
created: 2026-07-02
updated: 2026-07-02
tags: []
---
```

## Codex 应该做什么

Codex 最适合做的是“把半成品变成成品”。

### 适合的任务

- 把网页摘录整理成结构化笔记
- 把多篇碎片合并成一篇主题总结
- 从一组笔记里找出应该互相链接的内容
- 把项目相关笔记提炼成行动清单
- 根据周报/日记生成复盘
- 生成模板化内容，比如会议纪要、研究摘要、决策记录

### 不适合的任务

- 作为唯一知识存储
- 负责长期状态的权威来源
- 让它自由改写所有笔记而不保留来源
- 让它直接决定你的目录和标签体系

## 建议的 skills 切分

### 1. `capture-to-note`

输入：

- 网页片段
- 聊天记录
- 临时想法

输出：

- 一篇 Obsidian 笔记
- 包含标题、摘要、要点、来源、下一步

用途：

- 快速收集，不打断思路

### 2. `source-to-synthesis`

输入：

- 多篇来源笔记
- 一组阅读摘录

输出：

- 一篇主题总结
- 明确哪些是事实，哪些是你的判断

用途：

- 把碎片知识升维成可复用笔记

### 3. `link-suggester`

输入：

- 当前笔记
- 附近相关笔记列表

输出：

- 建议链接列表
- 每条链接的理由

用途：

- 减少笔记孤岛

### 4. `weekly-review`

输入：

- inbox
- active projects
- 最近新增笔记

输出：

- 待整理条目
- 待补充条目
- 下周行动项

用途：

- 保持知识库持续活着

### 5. `project-brief`

输入：

- 项目笔记集合

输出：

- 项目目标
- 关键资料
- 风险
- 下一步

用途：

- 让知识库直接服务执行

## 一个实用工作流

### 采集

1. 把网页、对话、灵感先丢进 `00_inbox`
2. Codex 用 `capture-to-note` 转成规范笔记
3. 写上来源和你自己的判断

### 归纳

1. 每天或每周跑一次 `weekly-review`
2. 把 inbox 里的内容分流到 `10_sources` / `20_atomic` / `40_projects`
3. 用 `source-to-synthesis` 合并同主题内容

### 连接

1. 用 `link-suggester` 找相关笔记
2. 人工确认链接
3. 只保留真正有意义的连接

### 使用

1. 做项目时，从知识库里拉资料
2. 用 `project-brief` 生成项目上下文
3. 让知识库直接输出到任务、文档、决策记录

## 组件边界

### Obsidian

- 存储
- 检索
- 双链
- 手工编辑

### Codex

- 结构化
- 总结
- 改写
- 提示链接
- 生成任务和模板

### skills

- 固定流程
- 标准输入输出
- 可重复执行

### MCP / 外部连接

- 拉外部知识源
- 不建议承担主存储职责

---

## 最小可行版本

如果你想最快开始，我建议只做这 4 件事：

1. 建好 vault 分层
2. 定义 1 套笔记模板
3. 先做 3 个 skills：`capture-to-note`、`weekly-review`、`project-brief`
4. 每周固定一次复盘

这个版本已经足够把知识库从“存笔记”推进到“持续产出”。

---

## 不建议的做法

- 一上来就做复杂数据库
- 标签数量失控
- 让 AI 自动重写全部历史笔记
- 只收集，不复盘
- 让知识库只服务阅读，不服务行动

## 结论

如果你的目标是“把 Obsidian 和 Codex 结合成个人知识库”，最靠谱的方式是：

- Obsidian 管存储
- Codex 管加工
- skills 管流程

先用轻量版跑起来，再逐步补自动化和索引层。这样最容易长期坚持，也最不容易把系统做复杂。
